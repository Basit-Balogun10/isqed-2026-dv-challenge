# Trace 05: Rampart I2C — Repeated START Creates STOP Condition

## Failing Assertion

```
ASSERTION FAILED: assert_rstart_no_stop
  Location: rampart_i2c_sva.sv:112
  Message: "Repeated START shall not generate an intermediate STOP condition on the bus"
  Failure time: cycle 400
```

## Spec Requirement

From the Rampart I2C Specification, Section 6.2.4:

> **Repeated START (Sr):** A repeated START condition is generated by the host
> without first issuing a STOP. The sequence is:
> 1. SCL goes low (from previous data/ACK phase)
> 2. SDA goes high while SCL is low (release SDA)
> 3. SCL goes high while SDA is high (setup phase)
> 4. SDA goes low while SCL is high (repeated START condition)
>
> CRITICAL: During step 2, SDA must transition high BEFORE SCL goes high.
> If SCL goes high while SDA is transitioning, it may be interpreted as a
> STOP condition (SDA rising while SCL is high) by target devices.

## Test Scenario

1. Configure I2C host: timing0_tlow=50, timing0_thigh=50, timing2_tsu_sta=10, timing2_thd_sta=10
2. Write address byte (0x50) with START, receive ACK
3. Write data byte (0xAA), receive ACK
4. Issue repeated START (Sr) to change direction (write-to-read)
5. Write address byte (0x50 | 0x01 = 0x51) for read
6. Verify no STOP condition appears on the bus during repeated START

## Observed Behavior

During the repeated START sequence, the host FSM enters HOST_RSTART_SETUP state.
In this state, the host releases both SDA and SCL to prepare for the new START.
However, the timing of the SDA and SCL transitions creates a brief window where
SDA rises while SCL is still high (from the previous ACK phase), which constitutes
a STOP condition on the I2C bus.

The root cause is in the HOST_RSTART_SETUP state logic: the host releases SDA
(sda_oe_o = 0, allowing pull-up to drive SDA high) at the same time it is supposed
to hold SCL low. But due to the timing counter starting from 0, SCL is released
one cycle before SDA is fully high, creating the SDA-rising-while-SCL-high pattern.
